**Motivacion:** Hoy el backend usa SQLite (`menteviva-backend/data/menteviva.db`). Para 100 usuarios concurrentes y multi-worker uvicorn, SQLite tira `database is locked`. Migramos a Postgres: en dev local via Docker (`docker-compose.dev.yml` ya creado), en prod via Neon (managed serverless).

El migrator del proyecto ya esta disenado portable (ver commit `6e7fa38: feat: Firebase Auth + DB Postgres-ready (schema migrator)`). Necesitamos validar que respeta `DATABASE_URL`, agregar el driver, y correr la primera migracion.

## Paso 3 — Refactor de connection

Reescribir `app/db/connection.py` (o equivalente) para soportar ambos backends segun `DATABASE_URL`:

```python
"""Pool de conexiones DB. Soporta SQLite (legacy) y Postgres (dev/prod).

DATABASE_URL examples:
- sqlite+aiosqlite:///data/menteviva.db        (default si no se setea env var)
- postgresql+psycopg://user:pass@host:5432/db  (dev Docker o prod Neon)
"""
import os
from typing import AsyncGenerator
import logging

logger = logging.getLogger("menteviva")

# Lazy import del driver segun la URL — evitar cargar psycopg cuando uso SQLite
# y viceversa.
def get_database_url() -> str:
    url = os.getenv("DATABASE_URL", "")
    if not url:
        # Default legacy: SQLite local
        from pathlib import Path
        db_path = Path(__file__).resolve().parent.parent.parent / "data" / "menteviva.db"
        db_path.parent.mkdir(parents=True, exist_ok=True)
        url = f"sqlite+aiosqlite:///{db_path}"
    return url


def is_postgres(url: str) -> bool:
    return url.startswith("postgresql")


# El resto del modulo depende del driver elegido. Si usamos un ORM (SQLAlchemy
# async), basta con create_async_engine(url). Si es raw SQL, hay que mantener
# dos paths.
#
# Recomendado: SQLAlchemy 2.x async para abstraer:
#   from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession
#   engine = create_async_engine(get_database_url(), echo=False, pool_pre_ping=True)
#
# Sin ORM, mantener funciones helper que delegan al driver correcto:
#   async def execute(sql: str, params: dict): ...
#   async def fetchall(sql: str, params: dict): ...
# Y dentro detectar postgres vs sqlite y usar psycopg o aiosqlite.
```

**Decision clave:** ¿usar SQLAlchemy o raw drivers?

- Si el proyecto actual usa raw `aiosqlite`, mantener raw y duplicar para `psycopg` es viable pero verbose. Mejor SQLAlchemy.
- Si ya usa SQLAlchemy, solo cambia la URL en el engine.

Auditar el codigo actual primero (paso 1) para decidir.

## Paso 4 — Adaptar el migrator

`app/db/migrator.py` debe:

1. Crear la tabla `schema_version` si no existe (SQL portable).
2. Leer la version actual.
3. Aplicar los `.sql` files que faltan en orden.
4. Bumpear la version.

El SQL `CREATE TABLE schema_version` ya deberia ser portable:

```sql
CREATE TABLE IF NOT EXISTS schema_version (
    version INTEGER PRIMARY KEY,
    applied_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

Riesgos: 
- `CURRENT_TIMESTAMP` en SQLite es local-timezone naive; en Postgres es `timestamp with time zone` por default. Si la columna se define explicitamente, hay que estandarizar a `TIMESTAMP WITHOUT TIME ZONE` o `TIMESTAMPTZ` consistente.
- Tipos numericos: `INTEGER` portable, `BIGSERIAL` solo Postgres (sustituir por `BIGINT GENERATED BY DEFAULT AS IDENTITY`).
- Indexes/constraints con sintaxis Postgres especifica (CASCADE, DEFERRABLE) — verificar que no existan.

Revisar TODOS los `.sql` files (probablemente 3-5 archivos) y validar:

```bash
cd menteviva-backend/app/db/migrations
grep -rE 'autoincrement|pragma|rowid|sqlite_' . || echo "OK, sin sqlite-isms"
```

## Paso 8 — DB branching para PRs (Neon feature)

Cuando hagamos un PR que toca el schema:

```bash
# Crear branch DB clone de main (no copia datos, COW)
neonctl branches create --project-id <proj> --name pr-<num>
# Output: connection string del branch

# Usarla en el .env del CI (GitHub Actions) o en preview deploy
# Borrar cuando se cierre el PR:
neonctl branches delete pr-<num>
```

Esto permite probar migrations destructivas con datos reales sin tocar prod.

## Riesgos / mitigaciones

| Riesgo | Mitigacion |
|---|---|
| El migrator actual NO esta tan portable como dice el commit | Auditar antes; si hay sqlite-isms, parchear los .sql y verificar idempotencia |
| Codigo de la app usa raw `lastrowid` o `LIMIT 1` no estandar | Reemplazar con `RETURNING id` (Postgres) o ORM agnostico |
| Locale Postgres dev vs Neon difieren -> ordenamientos cambian | Usar `LANG=es_MX.UTF-8` en docker-compose (ya esta) + verificar Neon region |
| `pg_isready` healthcheck del docker tarda en montar | El compose ya tiene retries=10. Si no, esperar manualmente con `until docker ps healthy` |
