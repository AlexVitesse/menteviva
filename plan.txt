# Plan: Personalizacion, Nivel Estricto y Mejora de Avatares

## Contexto
El sistema actual tiene prompts estaticos por avatar, sin niveles de dificultad ni personalizacion. Los avatares SVG son funcionales pero basicos (2 personajes). Se necesita:
1. Que el usuario pueda describir su situacion/producto para personalizar el escenario
2. Un nivel "cliente cerrado" que simule clientes dificiles y resistentes
3. Mejorar la calidad visual de los avatares

Con la incorporacion del prompt maestro de "Avatar Entrevistador por Competencias"
(BEI + STAR + Creswell) se agrega un paso 0 previo a las pruebas. Ese paso no solo
le da al usuario su mapa de brechas; tambien produce el CONTEXTO que hace que los
pasos 1, 2 y 3 sean realmente personalizados. Por eso el paso 0 va primero.

---

## 0. Fase Diagnostica Previa (PRIORIDAD 1)

### Por que va primero
Sin diagnostico, los tres puntos de abajo funcionan pero a medias:
- El "user_context" del paso 1 hoy es un textarea libre. El diagnostico produce un
  JSON estructurado que describe a la persona (no solo lo que vende).
- El nivel de dificultad del paso 2 se elige a ciegas. El diagnostico sugiere el
  nivel de arranque apropiado segun lo observado.
- Los avatares del paso 3 siguen siendo genericos. Con perfil inyectado, Roberto
  estresa las brechas especificas de este usuario (ej. "tiende a ceder rapido ante
  presion" -> Roberto presiona justo ahi).

Dicho de otra forma: paso 0 es el cableado que hace a 1-3 accionables.

### Flujo de producto
1. Usuario entra a la app sin nada guardado -> /registro (pantalla nueva,
   corta, ~5 campos de identidad, una sola vez)
2. Al terminar registro -> /diagnostico/setup (configuracion de esta sesion
   de diagnostico: idioma, tono, duracion, competencias foco opcionales)
3. /diagnostico: avatar entrevistador conduce 15-25 min de entrevista por
   competencias (5 fases: rapport, encuadre, desarrollo, profundizacion, cierre)
4. Al cerrar, se genera la parte "diagnostico" del user_profile (ver 0.3)
5. /diagnostico/perfil confirma al usuario lo encontrado -> dashboard
6. En cualquier Briefing posterior, el user_profile completo (registro +
   diagnostico) se inyecta en el system_prompt del avatar de prueba
7. El reporte post-prueba actualiza la parte "diagnostico" (merge, evolucion)

Separacion explicita:
- /registro = identidad permanente (una vez, re-editable desde /mi-plan)
- /diagnostico/setup = parametros de esta sesion (cada re-diagnostico)
- Esto se alinea con P-02 Registro de cuenta del spec original (CLAUDE.md)

### 0.1 Donde vive el prompt maestro

Opcion A (recomendada): nuevo avatar "entrevistador" en scenarios.py con el prompt
completo como system_prompt. Reutiliza la infra existente (WebSocket, STT, TTS,
streaming). Un solo lugar que mantener.

Opcion B: modulo aparte (app/diagnostic/) con router y analisis propios. Mas limpio
a largo plazo pero duplica plumbing (WS, auth, logging).

Decision: arrancar con A. Refactorizar a B solo si crece (ej. analisis de voz,
metricas de prosodia, estado de fases explicito).

### 0.2 Como entra el perfil en las pruebas (wiring clave)

Extender get_system_prompt() para aceptar user_profile opcional:

  def get_system_prompt(avatar_id, user_context="", difficulty="intermedio", user_profile=None):
      base = AVATARS[avatar_id]["system_prompt"]
      if user_profile:
          base += f"""

  CONTEXTO DEL USUARIO (de diagnostico previo):
  - Brechas a estresar: {user_profile['competencies_foco']}
  - Punto ciego: {user_profile['blind_spot']}
  - Tics verbales a cazar: {user_profile['verbal_patterns']['vague_verbs_detected']}
  Presiona especialmente esas dimensiones durante esta prueba.
  """
      if user_context:
          base += f"\n\nCONTEXTO DEL ESCENARIO:\n{user_context}"
      # ... luego aplicar DIFFICULTY_MODIFIERS del paso 2
      return base

Asi los pasos 1, 2 y 3 se apilan sin pelearse.

### 0.3 Contrato del user_profile (JSON)

El JSON se divide en dos bloques por origen: "registro" (lo teclea el usuario una
vez) y "diagnostico" (lo genera el LLM a partir de la conversacion). Asi queda claro
que edita el usuario directamente y que es derivado.

  {
    "user_id": "uuid-generado-client-side",
    "created_at": "ISO8601",
    "updated_at": "ISO8601",

    "registro": {
      "nombre": "Maria Lopez",
      "email": "maria@ejemplo.com",     // opcional, para migracion futura a DB
      "rol_objetivo": "Gerente de Ventas",
      "industria": "SaaS B2B",
      "experience_level": "entry|junior|mid|senior|lead|executive"
    },

    "diagnostico": {
      "completed_at": "ISO8601",
      "competencias_foco": ["manejo_objeciones", "defensa_valor"],
      "strengths": [
        { "skill": "comunicacion", "evidence": "cita textual",
          "why_matters": "..." }
      ],
      "gaps": [
        { "skill": "autoconciencia", "evidence": "...", "impact": "...",
          "micro_practice": "ejercicio accionable para esta semana" }
      ],
      "blind_spot": "usa 'nosotros' evasivo cuando se pide responsabilidad individual",
      "reflection_question": "...",
      "verbal_patterns": {
        "vague_verbs_detected": ["gestione", "manejo"],
        "we_vs_i_tendency": "alta|media|baja",
        "filler_frequency": "alta|media|baja"
      },
      "recommended_next_scenario": "roberto|maria|carlos",
      "recommended_next_level": "facil|intermedio|dificil"
    }
  }

Notas del contrato:
- "registro" se llena tras /registro y NO se toca despues salvo edicion manual.
- "diagnostico" se sobrescribe tras cada re-diagnostico. Reportes de pruebas hacen
  merge solo sobre strengths/gaps/blind_spot, no sobre recommended_next_*.
- Si "diagnostico" es null en el JSON, el guard de ruta manda a /diagnostico/setup.
- Variables del prompt maestro ({{nombre}}, {{rol}}, {{industria}}, {{nivel}}) se
  llenan directamente desde el bloque "registro". {{idioma}}, {{tono}}, {{minutos}},
  {{competencias}} se llenan desde /diagnostico/setup.

Este JSON es el contrato. Si se mantiene estable, el resto de la app puede evolucionar
independientemente (frontend puede sumar visualizaciones, backend puede cambiar
de modelo, analisis puede refinar heuristicas).

### 0.4 Persistencia (sin DB hoy)

Hasta que haya backend de usuarios:
- Frontend: localStorage key "menteviva:user_profile" con el JSON stringificado
  (un solo blob con registro + diagnostico, facil de migrar a una tabla)
- Al guardar /registro: se crea el JSON con "registro" lleno y "diagnostico": null.
- Al terminar /diagnostico: se rellena el bloque "diagnostico" (ver 0.3) y se
  actualiza "updated_at".
- Al conectar al WebSocket de una prueba, el frontend lee localStorage y manda el
  perfil en el primer mensaje (junto con user_context y difficulty del plan actual).
- Al terminar una prueba (avatar != entrevistador): merge parcial en "diagnostico"
  sobre strengths/gaps (acumulan con topN) y blind_spot (se reemplaza si hay evidencia
  nueva fuerte). recommended_next_* NO se toca en pruebas.
- Permitir "re-diagnostico" manual desde un boton en /mi-plan. Edicion manual del
  bloque "registro" tambien desde /mi-plan.

El contrato del 0.3 es identico a lo que ira en DB cuando exista, asi que la
migracion futura es trivial (1 tabla users con el blob JSON, o normalizado si
hace falta).

### 0.6 Consideraciones tecnicas

1. El prompt maestro pesa ~14 KB. llama-3.1-8b-instant tiene ~128k context, no satura,
   pero cada turno suma. Monitorear token count en logs.
2. Las 5 fases del prompt asumen que el LLM "sabe" en que fase esta. Sin estado
   explicito lo puede perder. Opciones en orden de complejidad:
   a) Dejar al LLM manejarlo (el prompt esta bien estructurado, puede funcionar)
   b) Inyectar "FASE_ACTUAL: N" en cada turno desde el backend
   c) State machine ligero en conversation.py que detecte transiciones
   Decision: arrancar con (a), caer a (b) si el LLM pierde el hilo en testing.
3. La salida del diagnostico es JSON estructurado, no texto libre. Reutilizar la
   estrategia ya probada en analyze_conversation(): modelo de razonamiento +
   "Responde UNICAMENTE con JSON valido".
4. Variables del prompt maestro ({{nombre}}, etc.) se llenan desde DiagnosticoSetup.
   Si el usuario deja algun campo vacio, el prompt lo infiere durante rapport (lo
   dice la seccion 1 del documento).
5. Detalle de idioma: el prompt maestro soporta es-MX / es-ES / en. Hoy groq_whisper
   fuerza language="es" en transcribe_audio. Habra que parametrizar si queremos
   soportar en.

### 0.7 Decisiones tomadas

- Registro: obligatorio una sola vez (sin password/auth en v1, solo datos
  basicos en localStorage). Re-editable desde /mi-plan.
- Diagnostico: obligatorio la primera vez (bloquea dashboard hasta completarlo).
  Re-diagnostico opcional desde /mi-plan cuando el usuario quiera.
- Persistencia: client-only via localStorage. Contrato listo para migrar a DB.
- Dashboard: muestra resumen del perfil (fortalezas top 2 + brecha principal).
  Vista completa en /mi-plan.
- Idioma: fijo es-MX para v1. groq_whisper ya fuerza language="es".
- Conversacion muy corta (<4 intercambios): reutilizar fallback existente de
  analyze_conversation ("necesitamos mas interacciones").
- UI de /registro: 5 campos, sin email obligatorio, sin password (no hay auth).

## 2. Nivel de Dificultad — Cliente Cerrado

### Problema
No hay variacion de dificultad. Roberto y Maria siempre se comportan igual.

### Solucion
Agregar selector de 3 niveles en Briefing que modifica el comportamiento del avatar via prompt.

### Niveles
| Nivel       | Nombre          | Comportamiento                                                              |
|-------------|-----------------|-----------------------------------------------------------------------------|
| facil       | Cooperativo     | Avatar abierto, objeciones suaves, da pistas                               |
| intermedio  | Neutral         | Comportamiento actual (default)                                             |
| dificil     | Cliente Cerrado | Resistente, impaciente, objeciones duras, interrumpe, amenaza con irse     |

### Archivos a modificar

**Frontend:**
- menteviva-frontend/src/types/index.ts — Agregar type DifficultyLevel = "facil" | "intermedio" | "dificil"
- menteviva-frontend/src/stores/sessionStore.ts — Agregar difficulty y setDifficulty (default: "intermedio")
- menteviva-frontend/src/pages/Briefing.tsx — Agregar selector de 3 botones de dificultad con iconos y descripcion
- menteviva-frontend/src/hooks/useWebSocket.ts — Enviar difficulty en mensaje de conexion
- menteviva-frontend/src/pages/Simulation.tsx — Pasar difficulty al hook

**Backend:**
- menteviva-backend/app/prompts/scenarios.py — Agregar DIFFICULTY_MODIFIERS dict con instrucciones por nivel:

  DIFFICULTY_MODIFIERS = {
      "facil": """
      NIVEL DE DIFICULTAD: COOPERATIVO
      - Se abierto y receptivo a la propuesta
      - Haz objeciones suaves que el vendedor pueda resolver facilmente
      - Da pistas sobre lo que necesitas
      - Muestra interes genuino
      """,
      "intermedio": "",  # Prompt base sin modificacion
      "dificil": """
      NIVEL DE DIFICULTAD: CLIENTE CERRADO
      - Eres muy dificil de convencer, casi imposible
      - Muestra desinteres activo: "No tengo tiempo", "No me interesa"
      - Interrumpe con objeciones duras antes de que termine de hablar
      - Amenaza con terminar la reunion: "Creo que no vamos a llegar a nada"
      - Solo se abre si el vendedor demuestra valor excepcional
      - Compara agresivamente con competencia: "Tu competidor me ofrece lo mismo mas barato"
      - Cuestiona todo: credibilidad, experiencia, resultados
      - Si el vendedor se pone nervioso o cede, presiona mas
      """
  }

- menteviva-backend/app/routers/conversation.py — Extraer difficulty del WebSocket y pasarlo a get_system_prompt()

## 3. Mejora de Avatares

### Problema actual
Los SVGs son funcionales pero basicos: formas simples, pocos detalles, expresiones limitadas.

### Mejoras propuestas

**A. Mas expresividad emocional** (en AnimatedAvatar.tsx)
- Agregar estado de "ceno fruncido" para nivel dificil (cejas inclinadas hacia abajo)
- Agregar expresion "sonrisa" para nivel facil (boca curvada hacia arriba)
- Agregar gesto de "mirar el reloj" cuando el avatar esta impaciente (nivel dificil)
- Variar la velocidad del parpadeo segun emocion (mas rapido = impaciente)

**B. Mejoras visuales del SVG**
- Agregar sombras mas suaves (drop-shadow filter en SVG)
- Mejorar degradados de piel con mas paradas de color
- Agregar fondos contextuales detras del avatar (oficina para Roberto, sala de juntas para Maria)
- Mejorar el cuello/hombros para que se vea mas profesional

**C. Tercer avatar: Carlos** (ya definido en backend como "carlos")
- Agregar configuracion de personaje en CHARACTERS de AnimatedAvatar.tsx
- Diseno: joven, casual, emprendedor tech
- Agregar su entry en scenarios.py con system prompt propio
- Escenario: Demo de producto tecnico (venta SaaS a startup)

### Archivos a modificar
- menteviva-frontend/src/components/avatar/AnimatedAvatar.tsx — Agregar expresiones emocionales, prop mood: "neutral" | "friendly" | "hostile", mejorar SVG, agregar Carlos
